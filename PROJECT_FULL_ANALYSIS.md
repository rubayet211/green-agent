## 1. Project Overview

**GreenAgent** is a multi-agent productivity copilot designed and developed for the **Beyond Tomorrow Summit 2026 Hackathon**. The application is structured around a novel concept: helping users optimize their daily focus and workspace efficiency while measuring and reducing their digital carbon footprint. 

The application utilizes a sequential workflow powered by Google Gemini (using the `gemini-3-flash-preview` model) to analyze a user's current digital workspace (number of open tabs, screen hours, active tasks, and work mode). It generates:
1. **Focus/Productivity Score** (indicating work habits efficiency).
2. **Digital Carbon Score** (behavioral estimate of network/energy impacts).
3. **Structured Recommendations** (curated, actionable advice).
4. **Primary Green Action Recommendation** (an optimized target behavior).

Users can select a recommendation and submit it to the **Hedera Consensus Service (HCS)** as a publicly verifiable "Green Action Log" proof. The system is designed to degrade gracefully: if Gemini API is unavailable, it switches automatically to a local deterministic calculation engine, and if Hedera credentials are not configured, it logs simulated proof records locally without generating fake transaction hashes.

## 5. Architecture Overview

GreenAgent is a full-stack Next.js web application implementing an API-centric model.

```
                      +-------------------+
                      |      Browser      |
                      +---------+---------+
                                |
                   Secure Cookie| (anonymousUserId)
                                v
                      +---------+---------+
                      |  Next.js Server   |
                      +---------+---------+
                                |
        +-----------------------+-----------------------+
        |                       |                       |
        v                       v                       v
+-------+-------+       +-------+-------+       +-------+-------+
|    lib/ai     |       | lib/firebase  |       |  lib/hedera   |
| (Gemini Chain)|       |  (Firestore)  |       |  (HCS Logs)   |
+-------+-------+       +---------------+       +---------------+
        |
        +-- (1) Context Analyzer
        +-- (2) Carbon Estimator
        +-- (3) Optimizer
        +-- (4) Action Recommender
```

### Server-Side Isolation
To protect API keys and database credentials, **Gemini orchestration**, **Firebase database connections**, and **Hedera network operations** are isolated in server-side libraries (`src/lib`). The client browser communicates with the server strictly via standard HTTP requests (`fetch` calls to `/api/*`).

### Signed Anonymous Identity Flow
To prevent spoofing and data leaks, GreenAgent uses a stateless cryptographic token structure:
1. When a user first interacts with the application, `ensureAnonymousIdentity()` calls `/api/identity`.
2. The server creates an anonymous `anonymousUserId` (UUID) and encapsulates it in a payload with an expiration time.
3. The payload is signed using HMAC-SHA256 with the server-only `ANONYMOUS_SESSION_SECRET`.
4. The token is appended to the response via a `Set-Cookie` header with `HttpOnly`, `SameSite=Lax`, and `Secure` (in production) flags.
5. Subsequent requests pass the cookie automatically, which the server parses and validates using constant-time comparisons (`crypto.timingSafeEqual`).

## 7. Product Purpose and Problem Solved

### The Problem
With the growth of modern remote work and cloud-based applications, browser multitasking has reached an all-time high. Digital workers frequently operate with dozens or hundreds of open browser tabs, sustained screen hours, and unstructured focus patterns. This creates two problems:
1. **Productivity Loss:** Constant context switching between tabs and tasks degrades attention span and decreases "Deep Work" phases.
2. **Digital Carbon Emissions:** Browser multitasking maintains persistent network synchronization loops, drives server-side memory allocations, increases client-side GPU rendering loads, and drains device batteries faster, indirectly raising energy footprint demand.

### The Solution
**GreenAgent** addresses both focus and digital emissions. It allows users to:
- Quickly input or select templates matching their current work session (tabs count, active hours, primary task objectives, and work mode).
- Receive immediate, concrete insights explaining the productivity risks of their context switching and the carbon drivers of their device energy use.
- Implement actionable recommendations (e.g., consolidating tabs, setting a digital blackout block, pausing auto-sync services) that improve concentration while lowering background network computing loads.
- Log their chosen improvements to a public blockchain network (Hedera Hashgraph) to document verifiable green focus habit milestones.

### User Roles
- **Anonymous Browser Profile:** GreenAgent does not require email registration or password authentication. Users are recognized instantly via their server-signed HTTP identity cookie. This keeps their metrics private to their browser and profile.

### Primary User Workflows

```
[User Form Entry] ---> [Multi-Agent Analysis] ---> [Dashboard Display]
                                                           |
                                                           v
[Explore Recommendations] <--- [Log Action to Hedera] <--- [Select Habit Action]
```

1. **Workspace Metric Entry:** The user lands on the Copilot page and logs current statistics (manually or by choosing a preset template like "Deep Work Day" or "Research Mode").
2. **Habit Optimization Analysis:** The user clicks "Analyze with GreenAgent". The system runs the analysis through the multi-agent orchestration pipeline.
3. **Metric Evaluation:** The user reviews their Focus and Carbon Scores (0–100 scale), evaluates the Context and Carbon narrative panels, and reads the primary recommended action.
4. **Action Commitment & Ledger Proof:** The user selects a specific recommendation from the list and clicks "Log as Green Action on Hedera".
5. **Consensus Verification:** The user views the receipt of the HCS submission, noting the real Consensus Time, Topic ID, and Transaction ID (with links to HashScan) or the clearly labeled simulation metadata.
6. **Habits History Trend Analysis:** The user clicks on the "History" tab to review their past logged sessions, tracking average Focus and Carbon score improvements over time.

## 9. Full Feature Inventory

- **Habit Input Console (`analysis-form.tsx`):** Range sliders for browser tabs (0-300), screen hours inputs (0-24), select triggers for work modes (Deep Work, Research, Meetings, Creative Work, Study), and key task list text areas. Includes validations ensuring inputs are bounded and non-empty.
- **Quick presets:** One-click presets ("Deep Work Day", "Research Mode", "Study Session", "Creative Work") that populate form fields instantly.
- **Multi-Agent AI Pipeline (`orchestrator.ts`):** Sequences four specialized agents in a dependency chain:
  1. *Context Analyzer:* Outlines productivity risks and work severity.
  2. *Carbon Estimator:* Highlights hardware and network drivers.
  3. *Optimizer:* Calculates focus/carbon scores and generates 3–4 practical recommendations.
  4. *Action Recommender:* Picks the single best suggestion.
- **Strict Schema Enforcement:** Every JSON output from the AI agents is checked against Zod schemas. If the AI output fails validation, the system retries (up to 2 times) and falls back gracefully to a local calculation engine.
- **Deterministic Local Fallback (`fallback.ts`):** Calculates scoring metrics and returns pre-validated, relevant suggestions if Gemini API is disabled or throws network timeout/rate-limit exceptions.
- **Hedera consensus logging (`client.ts`):** Connects to Hedera Testnet, Previewnet, or Mainnet and executes HCS Topic Message submissions containing session statistics. Returns consensus records.
- **Truthful Hedera Simulation:** Displays a detailed simulated receipt if no Hedera credentials are found, ensuring the application remains functional for demonstration without fabricating transaction hashes or ledger confirmation.
- **Anonymous Session History (`sessions.ts`):** Lists and retrieves owner-scoped sessions, rendering interactive summaries and average metric dashboards.
- **Cloud Firestore Adapter (`admin.ts`):** Handles secure server-side transactions with Google Firestore. Restricts fallback file persistence strictly to local development and test environments to protect production Vercel servers.

## 11. API / Backend Breakdown

GreenAgent endpoints are located under `src/app/api/` and utilize standard Next.js route handlers.

### 1. Identify Client Identity
- **Route:** `POST /api/identity`
- **Purpose:** Initializes or verifies the signed anonymous browser session token.
- **Behavior:** Parses existing identity cookie. If absent, signs a new cryptographically secure UUID token using the server's `ANONYMOUS_SESSION_SECRET` and sends it back in a `Set-Cookie` header.

### 2. Digital Habit Analysis
- **Route:** `POST /api/analyze`
- **Security:** Requires valid signed identity cookie. Enforces rate limits (maximum 5 calls per minute per identity).
- **Behavior:** Parses and validates input payload (tabs, hours, tasks, mode) using `AnalyzeRequestSchema`. Triggers `runAnalysis` which executes the four-agent Gemini sequence (or deterministic local engine). Wraps the results into a `GreenAgentSession` entity, persists it to the database, and returns the session object.

### 3. Log Action to Ledger
- **Route:** `POST /api/hedera/log-action`
- **Security:** Requires valid signed identity cookie. Enforces rate limits (maximum 5 calls per minute per identity).
- **Behavior:** Parses and validates the request (session ID, action ID). Retrieves the session record from database and confirms ownership. Matches the action ID to the session's recommendations. Submits a structured JSON message containing the action details to the configured HCS Topic. Updates the session's `selectedAction` and `hedera` metadata blocks, persists changes to the database, and returns the updated session object.

### 4. Fetch Session History
- **Route:** `GET /api/history`
- **Security:** Requires valid signed identity cookie.
- **Behavior:** Retrieves the authenticated user's session history, sorted in reverse-chronological order. Limits the returned documents to a maximum of 50 to optimize database reads.

### 5. Fetch Specific Session details
- **Route:** `GET /api/session/[id]`
- **Security:** Requires valid signed identity cookie.
- **Behavior:** Extracts the session ID from dynamic route parameters, retrieves the session record, and checks ownership. Returns a 404 error if the session does not exist or belongs to another user.

## 12. Database and Data Model Breakdown

### Database Orchestrator Layer (`sessions.ts`)
 GreenAgent uses a dual database adapter structure:
1. **Production Mode:** Employs the `firebase-admin` Firestore SDK. All session documents are saved to and queried from the Cloud Firestore database collection named `sessions`.
2. **Development / Testing Mode:** Fallback persistence using local filesystem JSON serializations (`data/local_sessions.json` or `data/local_sessions.test.json` in testing). If Firebase environment credentials are not configured, the system writes to these files instead of failing.
3. **Production Safety Gate:** In production environments (`NODE_ENV === "production"`), the file system fallback is strictly forbidden. If Firebase credentials are missing, the system throws a fatal error to prevent data loss or local storage leaks on serverless runtimes like Vercel.

### Database Query Indexing
The application uses a compound query filter inside `getHistory`:
```typescript
const snapshot = await firestore
  .collection("sessions")
  .where("anonymousUserId", "==", anonymousUserId)
  .orderBy("createdAt", "desc")
  .limit(limit)
  .get();
```
To support this query on Firestore, a composite index is configured in `firestore.indexes.json`:
- **Collection Name:** `sessions`
- **Fields:** `anonymousUserId` (Ascending) & `createdAt` (Descending)

### Data Schema Definition
The database schema is defined and validated by `GreenAgentSessionSchema` in `src/lib/validations/greenagent.ts`.

| Field Name | Type | Validation Rules | Description |
|---|---|---|---|
| **`id`** | string (UUID) | Validated UUID | Unique primary key of the session. |
| **`anonymousUserId`** | string (UUID) | Validated UUID | The signed browser identity of the session owner. |
| **`timestamp`** | string (ISO Date) | Validated DateTime | Timestamp when the habits analysis took place. |
| **`tabs`** | number (int) | Range: `0` to `300` | Uptime tab density of the browser. |

## 13. Authentication and Authorization

### Authentication Model
GreenAgent implements a cookie-based session strategy for browser-bound user tracking.

```
Browser                          Next.js Server
  |                                   |
  |--- POST /api/identity ----------->|
  |                                   | Parses & validates cookie signature.
  |                                   | If valid: returns 200.
  |                                   | If invalid/absent: creates UUID,
  |                                   | signs it, sets HttpOnly cookie.
  |<-- Response Cookie (ready: true) -|
```

- **Token Construction (`identity.ts`):** 
  Tokens are structured as standard base64url-encoded signed tokens: `[payload].[signature]`.
  - The payload contains: `{ id: anonymousUserId, expiresAt: timestamp + 1 year }`.
  - The signature is calculated as: `HMAC-SHA256(payload, ANONYMOUS_SESSION_SECRET)`.
- **Signature Verification:**
  - Token parameters are split into payload and signature.
  - An expected signature is calculated on the server and compared against the provided signature using constant-time comparison (`crypto.timingSafeEqual`) to prevent timing attacks.
  - The expiration date (`expiresAt`) is checked against the current server time.

### Authorization Checks
All database query paths protect user data ownership:
- **History Requests:** `getHistory` queries only documents matching the authenticated request's `anonymousUserId`.
- **Details Requests:** `getOwnedSession` checks the session record and returns `null` (causing a 404 error) if the `anonymousUserId` in the document does not match the requester.
- **Ledger Submission:** Logging an HCS transaction via `/api/hedera/log-action` checks ownership before executing the submission, preventing users from logging actions to sessions they do not own.

## 20. Weaknesses, Risks & Tech Debt

- **In-Memory Rate Limiting:** The rate limiter is currently in-memory. In a scaled production deployment with multiple server instances, rate limits will not be shared across servers.
  - *Fix:* Replace the in-memory map with a distributed rate limiter (e.g., Redis or Upstash).
- **Sequential AI Latency:** Running four sequential Gemini API calls can be slow (taking 3 to 6 seconds).
  - *Fix:* Optimize prompts or parallelize calls where possible, or stream the analysis progress to the client.
- **All-or-Nothing Fallback:** If any single Gemini agent in the sequence fails, the entire analysis falls back to the local calculation engine.
  - *Fix:* Implement partial-agent recovery so the chain can continue using previous successful outputs.
- **Client-Side Rendering Duplication:** The home page and the session detail page duplicate some recommendation rendering and Hedera logging logic.
  - *Fix:* Refactor the recommendation selection and Hedera submission logic into reusable React components.

## 22. Glossary of Important Internal Terms

- **HCS (Hedera Consensus Service):** A decentralized service on the Hedera network used to log messages and create immutable, time-stamped proofs.
- **Session:** A validated object representing a single workspace analysis, containing input parameters, calculated scores, AI outputs, and transaction details.

- **Fallback Engine:** The local calculation engine that provides deterministic scores and recommendations if the Gemini API is offline.
- **Anonymous Identity Token:** A signed token stored in an HttpOnly cookie that associates session records with a specific browser session.
